## ⚠️ Breaking Changes

| Version | Change | Migration |
|---------|--------|-----------|
| **v1.0** | Response format changed from raw `dict`s with semantic top-level `status` (`sent` / `scheduled` / `ok` / `cancelled`) to standardized `core.contracts.ok()` / `fail()` envelopes. Top-level `status` is now `"success"` or `"error"` only. The semantic status is preserved in `data.action_status` (`sent` / `scheduled` / `cancelled` / `ok` / `modified`). | Callers that branched on `result["status"] == "sent"` must now branch on `result["status"] == "success" and result["data"]["action_status"] == "sent"`. Callers that only checked `result["status"] == "error"` are unchanged. Update `docs/system_prompts/system_prompt.md` examples if any show the raw semantic `status` shape. |
| **v1.0** | `action` is now a required parameter and the `action: Literal["cancel","history","list","modify","recurring","schedule","send","test"]` enum is auto-generated by `@meta_tool`. Legacy calls that omitted `action` (rare) now return `{"status": "error", "error": "action is required (send | schedule | cancel | list | recurring | modify | history | test)"}`. | Pick the action that matches the intent — see the 8-action table in [API.md](API.md). |
| **v1.0** | New params `cron` (for `recurring`) and `trace_id` (observability) added to the facade signature. The `**kwargs` catch-all is preserved on handlers but the facade does not accept unknown kwargs (FastMCP schema rejects them). | Pure addition — no migration needed unless callers strict-validate the signature. |
| **v1.0** | Every response now includes a top-level `duration_ms` (added by the facade post-handler). | Pure addition — no migration needed unless callers strict-validate the response shape. |
| **v1.0** | `cancel` now fast-fails with `error_code="NOT_FOUND"` if `job_id` is not in `_job_registry` (previously it deferred to `scheduler.remove_job` and silently popped a missing key). | Callers that called `cancel` defensively on already-fired jobs must now check `error_code` instead of relying on silent success. The pre-v1 lenient behaviour is intentionally dropped because the registry is the authoritative source of truth (a job_id not in the registry either already fired or never existed). |
| **v1.0** | `list` returns `{"status": "success", "data": {"action_status": "ok", "jobs": [], "count": 0, "note": "Scheduler not running (APScheduler not installed)."}}` when APScheduler is missing (previously returned `{"status": "ok", "jobs": [], "count": 0, "note": "Scheduler not running"}`). | The envelope shape changed — semantic `ok` status moved to `data.action_status`. The "scheduler not running" case is still NOT an error (callers can call `list()` defensively). |
| **v1.0** | `_job_registry` is now persisted to `workspace/.notify_jobs/jobs.json` and reloaded on process startup. Jobs whose `run_at` is already in the past at reload time are skipped (their fire time has passed). | Pure addition — no migration. The persistence file is created lazily on first save; if it doesn't exist on startup, `_load_jobs()` returns silently. |

### 💡 Suggested Roadmap (Future Sessions)

The following items are **proposed** for future notify roadmap sessions. They are not yet committed — list them here so the next maintainer can pick the highest-value ones.

| Feature | Notes | Priority |
|---------|-------|----------|
| `notify(action="ntfy")` | Send via [ntfy.sh](https://ntfy.sh) (self-hosted Docker push service; HTTP POST → push to phone). Enables remote / headless notifications. Likely needs `backend` + `backend_config={"topic": "..."}` params (see "Delivery backend abstraction" below). | P2 |
| `notify(action="email")` | Send via SMTP. Recipient-scoped notification. Requires SMTP config in `.env` (`SMTP_HOST`, `SMTP_USER`, `SMTP_PASS`). | P2 |
| `notify(action="webhook")` | Send via HTTP POST to a URL. Generic integration — caller supplies URL + optional headers + JSON body template. | P3 |
| `notify(action="slack")` | Send via Slack incoming webhook. Channel-scoped notification. Requires `SLACK_WEBHOOK_URL` in `.env`. | P2 |
| `notify(action="discord")` | Send via Discord incoming webhook. Bot-channel notification. Requires `DISCORD_WEBHOOK_URL` in `.env`. | P2 |
| `notify(action="telegram")` | Send via Telegram Bot API. Chat-scoped notification. Requires `TELEGRAM_BOT_TOKEN` + `TELEGRAM_CHAT_ID` in `.env`. | P2 |
| `notify(action="quiet_hours")` | Set do-not-disturb window (e.g. `start="22:00"`, `end="07:00"`). Notifications during quiet hours are silently queued (or routed to a non-desktop backend). | P3 |
| `notify(action="priority")` | Set notification priority level (`low` / `default` / `high` / `urgent`). High priority could bypass quiet hours or route to a more visible backend (Slack instead of desktop). | P3 |
| `notify(action="batch")` | Send multiple notifications in one call. Reduces facade round-trips when alerting on N events. Spec: `notifications=[{title, message, ...}, ...]`. | P3 |
| Notification templates | Pre-configured `title`/`message` pairs for common alerts (e.g. `"build_failed"`, `"test_passed"`, `"deadline_warning"`). Callers reference by template name + supply dynamic values. | P3 |
| Delivery backend abstraction | Pluggable backends (desktop / ntfy / slack / discord / telegram / email) via a `_BACKENDS` dict in `notify_ops/backends.py`. Each backend is a callable `(title, message, config) -> (success, method)`. Send/test actions accept a `backend` param (default `"desktop"`). Future `schedule` tool can route through the same backend abstraction. | P2 |
| Separate `schedule` tool | Cron jobs, calendar sync (iCal / CalDAV), recurring task execution. The `schedule` tool will OWN the scheduling logic and USE `notify(action="send")` as its delivery mechanism — notify stays focused on notification delivery, not scheduling logic beyond simple delays + cron. See [recurring.py source](https://github.com/) docstring for the proposed `schedule(action="add_cron", cron=..., delivery=notify(action="send", ...))` API. | P3 |
| `notify` as delivery backend for `schedule` tool | The future `schedule` tool calls `notify(action="send", title="...", message="...")` when a scheduled task fires. Notify should NOT grow scheduling logic — that's `schedule`'s job. The current `recurring` action is a stop-gap until `schedule` lands. | P3 |
| `modify` v1.1 — reschedule APScheduler job | *(deduplicated — listed in In Progress above)*. Make `modify` actually update the next-fire payload by changing the firing callback to look up `_job_registry` by `job_id` at fire time. | P2 |

> **Note for future maintainers:** items in the table above are *suggestions* gathered during v1.0 docs work. Before implementing any of them, re-check the current source (`tools/notify_ops/`) and the test layout (`tests/tools/notify/`) to confirm the refactor surface. The `schedule` tool split in particular should be its own staging refactor — moving `recurring` (and possibly `schedule`) out of notify is a non-trivial API break.

## 🚫 Deferred / Out of Scope

| # | Feature | Why Deferred | Priority |
|---|---------|------------|----------|
| 1 | **Persistent database backend for jobs** | In-memory + JSON file is sufficient. Full DB overkill for notification scheduling. JSON file gives restart-survival without the operational overhead of a DB. | Skip |
| 2 | **Web UI for managing notifications** | Out of scope. Use `action="list"` and `action="cancel"` via tool calls. A web UI belongs in the agent's UI layer, not the notify tool. | Skip |
| 3 | **Rich media attachments** | Desktop notification APIs have limited support (no images in `notify-send`; plyer attachment support is patchy). Out of scope until a backend that supports media (Slack, Discord, Telegram) lands. | Skip |
| 4 | **Persistent audit trail of delivered notifications** | The in-memory `_delivery_log` (capped at 50) is a debugging aid, NOT an audit trail. If you need a persistent audit trail, wire up a delivery backend (ntfy.sh / Slack / Discord / Telegram / email) which has its own server-side history. Cross-process state coupling is explicitly avoided. | Skip |
| 5 | **SMS notifications** | Out of scope for notify tool. SMS requires a paid provider (Twilio, etc.) and belongs in a dedicated integration, not the notify meta-tool. Use `notify(action="webhook")` (proposed) to bridge to an external SMS gateway. | Skip |
| 6 | **Migrate `recurring` action to the future `schedule` tool** | Once `schedule` lands, `recurring` should be deprecated (or re-exported as a thin shim that calls `schedule(action="add_cron", delivery=notify(action="send", ...))`). Until then, `recurring` stays in notify as a stop-gap. | Deferred (waits on `schedule` tool) |
