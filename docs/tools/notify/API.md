# 📝 API Reference

## 🔧 Tool Signature

```python
@tool
@meta_tool(
    DISPATCH.get("notify", {}),
    doc_sections=[
        "NOTIFY TOOL — Desktop notifications + scheduled reminders:",
        " | Need | Action | Why |",
        " |------|--------|-----|",
        " | Immediate notification | notify(send) | Desktop alert (plyer/notify-send/console) |",
        " | Schedule a reminder | notify(schedule) | APScheduler DateTrigger after N minutes |",
        " | Cancel a reminder | notify(cancel) | Remove job by job_id |",
        " | List scheduled jobs | notify(list) | Show all pending notifications |",
        " | Recurring notification | notify(recurring) | Cron-style via APScheduler CronTrigger |",
        " | Modify a job | notify(modify) | Update title/message without cancel+re-create |",
        " | Recently sent | notify(history) | In-memory log of delivered notifications |",
        " | Test delivery | notify(test) | Verify notification pipeline works |",
        "",
        "Platform: Windows (plyer), Linux (notify-send), fallback (console).",
        "PARALLEL_SAFE — notifications are stateless.",
        "",
        "BREAKING v1.0: response format standardized via ok()/fail().",
        " - response.status is 'success' or 'error' (was 'sent'/'scheduled'/'cancelled'/'ok').",
        " - Semantic status preserved in data.action_status.",
        " - duration_ms added to every response.",
    ],
)
def notify(
    action: str = "",
    title: str = "",
    message: str = "",
    timeout: int = 5,
    delay_minutes: int = 0,
    job_id: str = "",
    cron: str = "",
    trace_id: str = "",
    **kwargs,
) -> dict:
    """notify meta-tool — send | schedule | cancel | list | recurring | modify | history | test."""
```

The `action: Literal["cancel","history","list","modify","recurring","schedule","send","test"]` type annotation is **auto-generated by `@meta_tool`** from `DISPATCH["notify"].keys()` at decoration time. The docstring is also auto-generated from each action's `help_text` + `examples` + the `doc_sections` list above.

### Parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `action` | `str` | **Yes** | `""` | One of `send` \| `schedule` \| `cancel` \| `list` \| `recurring` \| `modify` \| `history` \| `test`. Empty string returns `{"status": "error", "error": "action is required (send | schedule | cancel | list | recurring | modify | history | test)"}`. Unknown values return `"Unknown action 'X'. Use: cancel | history | list | modify | recurring | schedule | send | test"`. Case-insensitive (lowercased internally). |
| `title` | `str` | No | `""` | Notification title. Default applied per action: `"Agent"` for `send`/`test`; `"Agent Reminder"` for `schedule`/`recurring`. Ignored by `cancel`/`list`/`history`. |
| `message` | `str` | No | `""` | Notification body. **Required** for `send`, `schedule`, `recurring`. Ignored by `cancel`/`list`/`history`/`test`. |
| `timeout` | `int` | No | `5` | Display duration in seconds. Used by `send` only (passed to `plyer`/`notify-send -t`). Ignored by other actions. |
| `delay_minutes` | `int` | No | `0` | Delay before scheduled notification (minutes). **Required** and must be `> 0` for `schedule`. Ignored by other actions. |
| `job_id` | `str` | No | `""` | Job identifier (from `schedule`/`recurring` response). **Required** for `cancel` and `modify`. Ignored by other actions. |
| `cron` | `str` | No | `""` | 5-field cron expression (e.g. `"0 9 * * *"` = 9am daily). **Required** for `recurring`. Ignored by other actions. |
| `trace_id` | `str` | No | `""` | Trace identifier for logging and result correlation. Threaded into every response (top-level when non-empty, plus in `data.trace_id`). |

## 📤 Response Envelope (BREAKING v1.0)

All responses are `ToolResult` dicts from `core.contracts`. The facade adds `duration_ms` to every handler-returned response.

### Success

```json
{
  "status": "success",
  "trace_id": "abc123",
  "data": {
    "action_status": "sent",
    "action": "send",
    "title": "Research done",
    "message": "Tesla analysis complete",
    "method": "plyer",
    "trace_id": "abc123"
  },
  "duration_ms": 12
}
```

### Error

```json
{
  "status": "error",
  "trace_id": "abc123",
  "error": "message is required for send",
  "error_code": "MISSING_PARAM",
  "data": null,
  "duration_ms": 1
}
```

### Facade-level errors (no `duration_ms`)

Empty `action` and unknown `action` are returned BEFORE the timing block — they don't get `duration_ms`:

```json
{
  "status": "error",
  "data": null,
  "error": "action is required (send | schedule | cancel | list | recurring | modify | history | test)",
  "trace_id": ""
}
```

### `data.action_status` — semantic status (preserved from pre-v1)

| `action_status` | Meaning | Actions |
|-----------------|---------|---------|
| `sent` | Immediate notification delivered | `send`, `test` |
| `scheduled` | Reminder queued for future delivery | `schedule`, `recurring` |
| `cancelled` | Scheduled job removed | `cancel` |
| `modified` | Job metadata updated in registry | `modify` |
| `ok` | Query/list operation succeeded | `list`, `history` |

> **BREAKING v1.0 migration:** pre-v1 callers that branched on `result["status"] == "sent"` must now branch on `result["status"] == "success" and result["data"]["action_status"] == "sent"`. Callers that only checked `result["status"] == "error"` are unchanged.

## 🎬 Actions

### `action="send"` — Immediate desktop notification

Sends a desktop notification immediately using the cross-platform fallback chain.

**Params used:** `title`, `message`, `timeout`, `trace_id`.

**Fallback chain:**
1. Windows → `plyer.notification.notify()` (native toast)
2. Linux → `notify-send` (libnotify) — only if `plyer` failed or we're not on Windows
3. Universal → `print()` to `sys.stderr`

**Returns:** `ok({"action_status": "sent", "action": "send", "title": str, "message": str, "method": "plyer"|"notify-send"|"console", "trace_id": str}, trace_id=trace_id)`

| Field | Type | Description |
|-------|------|-------------|
| `data.action_status` | `str` | `"sent"` |
| `data.action` | `str` | `"send"` |
| `data.title` | `str` | Resolved title (default: `"Agent"`) |
| `data.message` | `str` | Notification body |
| `data.method` | `str` | Delivery method used: `"plyer"`, `"notify-send"`, or `"console"` |
| `data.trace_id` | `str` | Trace identifier (echoed) |

**Examples:**
```python
notify(action="send", title="Research done", message="Tesla analysis complete")
notify(action="send", message="Build finished", trace_id="run-42")
```

---

### `action="schedule"` — Schedule a one-shot delayed notification

Queues a notification for future delivery via APScheduler `DateTrigger` (N minutes from now).

**Params used:** `title`, `message`, `delay_minutes`, `trace_id`.

**Config requirement:** `apscheduler` must be installed (returns `DEPENDENCY_MISSING` error otherwise).

**Returns:** `ok({"action_status": "scheduled", "action": "schedule", "job_id": str, "message": str, "run_at": str, "delay_minutes": int, "trace_id": str}, trace_id=trace_id)`

| Field | Type | Description |
|-------|------|-------------|
| `data.action_status` | `str` | `"scheduled"` |
| `data.action` | `str` | `"schedule"` |
| `data.job_id` | `str` | Unique identifier (format: `reminder_<unix_timestamp>`) — use with `cancel` |
| `data.message` | `str` | Notification body |
| `data.run_at` | `str` | Formatted `"YYYY-MM-DD HH:MM:SS"` scheduled time |
| `data.delay_minutes` | `int` | Original delay requested |
| `data.trace_id` | `str` | Trace identifier (echoed) |

**Side effects:** Adds APScheduler job + updates `_job_registry` + calls `_save_jobs()` (persists to `workspace/.notify_jobs/jobs.json`).

**Examples:**
```python
notify(action="schedule", message="Check autocode results", delay_minutes=10)
notify(action="schedule", title="Standup", message="9am standup", delay_minutes=60, trace_id="d-1")
```

---

### `action="cancel"` — Cancel a scheduled or recurring notification

Removes a scheduled job from APScheduler and the in-memory registry.

**Params used:** `job_id`, `trace_id`.

**Returns:** `ok({"action_status": "cancelled", "action": "cancel", "job_id": str, "trace_id": str}, trace_id=trace_id)`

**Side effects:** Calls `scheduler.remove_job(job_id)` (silently caught if already fired) + pops from `_job_registry` + calls `_save_jobs()`.

**Examples:**
```python
notify(action="cancel", job_id="reminder_1234567890")
notify(action="cancel", job_id="recurring_1234567890", trace_id="cleanup-1")
```

---

### `action="list"` — List all scheduled and recurring notifications

Returns all jobs currently in the APScheduler job store, enriched with registry metadata.

**Params used:** `trace_id` only.

**Returns:** `ok({"action_status": "ok", "action": "list", "jobs": [...], "count": int, "trace_id": str}, trace_id=trace_id)`

When the scheduler is `None` (APScheduler not installed), returns `ok({"action_status": "ok", "action": "list", "jobs": [], "count": 0, "note": "Scheduler not running (APScheduler not installed).", "trace_id": str}, trace_id=trace_id)` — NOT an error so callers can call `list()` defensively.

**Per-job entry shape:**
```json
{
  "job_id": "reminder_1234567890",
  "run_at": "2026-07-15 18:00:00",
  "title": "Agent Reminder",
  "message": "Check autocode results",
  "recurring": false
}
```

For recurring jobs, an additional `cron` field is included:
```json
{
  "job_id": "recurring_1234567890",
  "run_at": "2026-07-15 09:00:00",
  "title": "Standup",
  "message": "Daily standup time",
  "recurring": true,
  "cron": "0 9 * * *"
}
```

**Examples:**
```python
notify(action="list")
notify(action="list", trace_id="audit-1")
```

### `action="recurring"` — Schedule a cron-style recurring notification

Queues a recurring notification via APScheduler `CronTrigger.from_crontab()`.

**Params used:** `title`, `message`, `cron`, `trace_id`.

**Config requirement:** `apscheduler` must be installed (returns `DEPENDENCY_MISSING` error otherwise).

**Returns:** `ok({"action_status": "scheduled", "action": "recurring", "job_id": str, "cron": str, "next_run": str, "title": str, "message": str, "trace_id": str}, trace_id=trace_id)`

| Field | Type | Description |
|-------|------|-------------|
| `data.action_status` | `str` | `"scheduled"` (same as `schedule` — both queue a future fire) |
| `data.action` | `str` | `"recurring"` |
| `data.job_id` | `str` | Unique identifier (format: `recurring_<unix_timestamp>`) — use with `cancel` |
| `data.cron` | `str` | The cron expression (echoed, stripped) |
| `data.next_run` | `str` | Formatted `"YYYY-MM-DD HH:MM:SS"` of the next fire time (computed via `trigger.get_next_fire_time(None, now)`) |
| `data.title` | `str` | Resolved title (default: `"Agent Reminder"`) |
| `data.message` | `str` | Notification body |
| `data.trace_id` | `str` | Trace identifier (echoed) |

**Side effects:** Adds APScheduler CronTrigger job + updates `_job_registry` (with `recurring=True` + `cron` field) + calls `_save_jobs()`.

#### Cron expression syntax

Standard Unix 5-field cron (APScheduler's `CronTrigger.from_crontab` matches vixie-cron semantics):

```
┌──────── minute (0-59)
│ ┌────────── hour (0-23)
│ │ ┌──────────── day-of-month (1-31)
│ │ │ ┌────────────── month (1-12)
│ │ │ │ ┌──────────────── day-of-week (0-6 where 0=Sunday)
│ │ │ │ │
│ │ │ │ │
* * * * *
```

| Expression | Meaning |
|------------|---------|
| `*/5 * * * *` | every 5 minutes |
| `0 9 * * *` | 9am daily |
| `0 9 * * 1` | 9am every Monday |
| `0 0 1 * *` | midnight on the 1st of every month |
| `0 */2 * * *` | every 2 hours on the hour |
| `30 8 * * 1-5` | 8:30am Monday through Friday |

> **Note:** APScheduler's `from_crontab` does NOT support `@yearly`/`@monthly`/`@weekly`/`@daily`/`@hourly`/`@reboot` macros. Use the equivalent 5-field expression. CRON_TZ (per-job timezone) is also not supported in the current `recurring` action — planned for the future `schedule` tool.

**Examples:**
```python
notify(action="recurring", cron="0 9 * * *", title="Standup", message="Daily standup time")
notify(action="recurring", cron="*/15 * * * *", message="Heartbeat")
notify(action="recurring", cron="0 0 1 * *", title="Monthly review", message="Time for the monthly code review", trace_id="cron-1")
```

### `action="modify"` — Update an existing job's title and/or message

Updates the `_job_registry` metadata for an existing scheduled or recurring job WITHOUT canceling and re-creating it.

**Params used:** `job_id`, `title` (optional), `message` (optional), `trace_id`.

**Returns:** `ok({"action_status": "modified", "action": "modify", "job_id": str, "updated_fields": [str], "title": str, "message": str, "trace_id": str}, trace_id=trace_id)`

| Field | Type | Description |
|-------|------|-------------|
| `data.action_status` | `str` | `"modified"` |
| `data.action` | `str` | `"modify"` |
| `data.job_id` | `str` | Job identifier (echoed) |
| `data.updated_fields` | `list[str]` | Which fields were updated: subset of `["title", "message"]` |
| `data.title` | `str` | Updated title (current registry value) |
| `data.message` | `str` | Updated message (current registry value) |
| `data.trace_id` | `str` | Trace identifier (echoed) |

**Side effects:** Updates `_job_registry[job_id]` in place + calls `_save_jobs()`.

#### ⚠️ v1.0 Limitation — metadata only, NOT reschedule

APScheduler jobs reference `_send_notification` at scheduling time, with `title`/`message` baked in as `kwargs`. The current implementation passes them as kwargs, so they are NOT re-read from the registry on fire. This means:

- **`modify` updates the metadata shown by `list()` immediately** ✅
- **`modify` does NOT change the title/message of the NEXT fire** ❌ (the APScheduler job still has the old kwargs baked in)

To make a change take effect on the next fire, use `cancel` + `schedule` (or `cancel` + `recurring`):

```python
# v1.0 workaround for "modify the next fire's payload"
notify(action="cancel", job_id="reminder_123")
notify(action="schedule", title="New title", message="New body", delay_minutes=10)
```

**v1.1 plan:** Change the firing callback to look up `_job_registry` by `job_id` at fire time, so metadata changes propagate to the next fire without cancel+re-create. Tracked in [CHANGELOG.md → In Progress](CHANGELOG.md#-in-progress--next-up).

**Examples:**
```python
notify(action="modify", job_id="reminder_123", title="Updated Title")
notify(action="modify", job_id="recurring_456", title="New", message="New body")
notify(action="modify", job_id="reminder_789", message="Just the body changes", trace_id="edit-1")
```

### `action="history"` — Show recently sent notifications

Returns the last 20 entries from the in-memory delivery log (`state._delivery_log`).

**Params used:** `trace_id` only.

**Returns:** `ok({"action_status": "ok", "action": "history", "notifications": [...], "count": int, "total_logged": int, "trace_id": str}, trace_id=trace_id)`

| Field | Type | Description |
|-------|------|-------------|
| `data.action_status` | `str` | `"ok"` |
| `data.action` | `str` | `"history"` |
| `data.notifications` | `list[dict]` | Last 20 delivery log entries (newest last) |
| `data.count` | `int` | Number of entries returned (≤ 20) |
| `data.total_logged` | `int` | Total entries in the full log (capped at 50) |
| `data.trace_id` | `str` | Trace identifier (echoed) |

**Per-entry shape** (appended by `_log_delivery()` after every `_send_notification`):
```json
{
  "title": "Research done",
  "message": "Tesla analysis complete",
  "method": "plyer",
  "trace_id": "abc123",
  "timestamp": "2026-07-15T14:30:45.123456"
}
```

> **Note:** The log is in-memory only and is NOT persisted across process restarts. It is bounded to 50 entries (`_MAX_DELIVERY_LOG`) — older entries drop off the front. The action returns the last 20 (not the full 50) to keep the response payload small.

**Examples:**
```python
notify(action="history")
notify(action="history", trace_id="audit-1")
```

---

### `action="test"` — Send a test notification

Sends a fixed test notification through the full `_send_notification` pipeline so the LLM (or a human operator) can verify the delivery chain works end-to-end without crafting a real notification payload.

**Params used:** `trace_id` only.

**Returns:** `ok({"action_status": "sent", "action": "test", "title": "Test", "message": "Notification test successful", "method": str, "trace_id": str}, trace_id=trace_id)`

The test notification uses fixed `title="Test"` and `message="Notification test successful"` so it's identifiable in the delivery log (`history` action) and distinguishable from real notifications.

**Examples:**
```python
notify(action="test")
notify(action="test", trace_id="smoke-1")
```

## 🔒 Security

Current considerations:
- `subprocess.run(["notify-send", "-t", str(timeout * 1000), title, message], capture_output=True, timeout=5)` is called with a hardcoded argv list — no shell, no shell injection risk. The `title` and `message` are passed as separate argv elements (not interpolated into a shell string).
- Job IDs are generated from `int(time.time())` — not cryptographically random, but sufficient for notification scheduling. Do NOT reuse this pattern for security tokens.
- The persistence file `workspace/.notify_jobs/jobs.json` is written with default filesystem permissions (umask). On multi-user systems, consider tightening permissions if the message bodies are sensitive.
- The cron expression is validated by `CronTrigger.from_crontab()` BEFORE the job is added — malformed expressions return `INVALID_PARAM` rather than crashing the scheduler.

## 🚨 Error Handling

| Error source | Where raised | Envelope `status` | `error_code` | Example |
|--------------|-------------|-------------------|--------------|---------|
| Empty `action` | Facade | `"error"` (no `duration_ms`) | — | `"action is required (send \| schedule \| cancel \| list \| recurring \| modify \| history \| test)"` |
| Unknown `action` | Facade | `"error"` (no `duration_ms`) | — | `"Unknown action 'X'. Use: cancel \| history \| list \| modify \| recurring \| schedule \| send \| test"` |
| Missing `message` (send/schedule/recurring) | Action handler | `"error"` | `MISSING_PARAM` | `"message is required for send"` |
| Missing or non-positive `delay_minutes` (schedule) | Action handler | `"error"` | `INVALID_PARAM` | `"delay_minutes must be > 0 for schedule"` |
| Missing `job_id` (cancel/modify) | Action handler | `"error"` | `MISSING_PARAM` | `"job_id is required for cancel"` |
| Missing/whitespace `cron` (recurring) | Action handler | `"error"` | `MISSING_PARAM` | `"cron is required for recurring (e.g. '0 9 * * *' = 9am daily)"` |
| Invalid cron expression (recurring) | Action handler | `"error"` | `INVALID_PARAM` | `"Invalid cron expression '99 * * * *': ..."` |
| Job not found in registry (cancel/modify) | Action handler | `"error"` | `NOT_FOUND` | `"Job 'X' not found in registry (it may have already fired or never existed)."` |
| No fields provided (modify) | Action handler | `"error"` | `MISSING_PARAM` | `"At least one of title or message must be provided for modify."` |
| APScheduler not installed (schedule/cancel/recurring) | Action handler | `"error"` | `DEPENDENCY_MISSING` | `"APScheduler not installed. Run: pip install apscheduler"` |
| Scheduler not running (cancel) | Action handler | `"error"` | `DEPENDENCY_MISSING` | `"Scheduler not running (APScheduler not installed)."` |
| Schedule/recurring add_job failure | Action handler | `"error"` | `INTERNAL_ERROR` | `"Schedule failed: <exception>"` / `"Recurring schedule failed: <exception>"` |
| List failure (scheduler.get_jobs raises) | Action handler | `"error"` | `INTERNAL_ERROR` | `"List failed: <exception>"` |
| Delivery failure (send/test) | Action handler | `"error"` | `DELIVERY_FAILED` | `"Notification delivery failed for method=console"` / `"Test notification delivery failed for method=console"` |
| Handler raises unexpected exception | Facade try/except | `"error"` (no `duration_ms`) | — | `"Notify action failed: <exception message>"` |
| Handler returns non-dict | Facade | `"error"` (no `duration_ms`) | — | `"Handler returned <type>, expected dict."` |

### Per-action error semantics summary

| Action | On missing param | On APScheduler missing | On job not found |
|--------|------------------|------------------------|------------------|
| `send` | `MISSING_PARAM` (message required) | N/A — `send` does not use APScheduler | N/A |
| `schedule` | `MISSING_PARAM` (message) / `INVALID_PARAM` (delay) | `DEPENDENCY_MISSING` | N/A |
| `cancel` | `MISSING_PARAM` (job_id) | `DEPENDENCY_MISSING` (scheduler not running) | `NOT_FOUND` (fast-fail before scheduler.remove_job) |
| `list` | N/A | Returns empty list with note (NOT error) | N/A |
| `recurring` | `MISSING_PARAM` (message / cron) / `INVALID_PARAM` (bad cron) | `DEPENDENCY_MISSING` | N/A |
| `modify` | `MISSING_PARAM` (job_id / no fields) | N/A — `modify` does not use APScheduler | `NOT_FOUND` |
| `history` | N/A | N/A — `history` does not use APScheduler | N/A |
| `test` | N/A | N/A — `test` does not use APScheduler | N/A |

## 🔮 Future API Extensions (NOT yet implemented)

The following are proposed extensions for future notify versions. They are documented here so callers can plan for them — see [CHANGELOG.md → Suggested Roadmap](CHANGELOG.md#-suggested-roadmap-future-sessions) for the full list with priorities.

### Delivery backend selection (proposed v2.0+)

```python
# Send via ntfy.sh instead of desktop
notify(action="send", title="...", message="...",
       backend="ntfy", backend_config={"topic": "agent-alerts"})

# Send via Slack
notify(action="send", title="...", message="...",
       backend="slack", backend_config={"webhook_url": "..."})
```

Proposed backends: `desktop` (current default), `ntfy`, `slack`, `discord`, `telegram`, `email`, `webhook` (generic HTTP POST).

### Separate `schedule` tool (proposed)

A future `schedule` tool will own richer scheduling semantics (iCal/CalDAV sync, CRON_TZ, human-readable "every weekday" parsing, recurring task execution that's NOT just notification). It will USE `notify(action="send")` as its delivery mechanism:

```python
# Future API (proposed — not yet implemented):
schedule(action="add_cron", cron="0 9 * * *",
         delivery=notify(action="send", title="Standup", message="Daily standup time"))
```

When `schedule` lands, notify's `recurring` action will likely be deprecated (or re-exported as a thin shim).
