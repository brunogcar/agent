# 📝 API Reference

## 🔧 Tool Signature

```python
@tool
@meta_tool(
    DISPATCH.get("parallel", {}),
    doc_sections=[
        "PARALLEL TOOL — Execute multiple tool calls concurrently:",
        " | Need | Action | Why |",
        " |------|--------|-----|",
        " | All tools in parallel, wait for all | parallel(run) | ThreadPoolExecutor, wait for completion |",
        " | First successful result wins | parallel(race) | as_completed, cancel rest on first success |",
        " | Sequential chain (output feeds next) | parallel(pipeline) | NOT parallel — ordered execution |",
        "",
        "Each task is a dict: {name: str, args: dict}",
        "PARALLEL_SAFE tools: web, file, python, python_exec, notify, github, consult, vision, report, agent",
        "NOT parallel-safe: cli, git, memory, browser, tavily, swarm, workflow, parallel",
        "",
        "BREAKING v1.0: param `tools` renamed to `tasks`. action is now required.",
    ],
)
def parallel(
    action: str = "",
    tasks: list[dict] = [],
    max_workers: int = 4,
    allow_unsafe: bool = False,
    timeout: int = -1,
    trace_id: str = "",
) -> dict:
    """Parallel tool execution meta-tool — run | race | pipeline."""
```

The `action: Literal["pipeline", "race", "run"]` type annotation is **auto-generated by `@meta_tool`** from `DISPATCH["parallel"].keys()` at decoration time. The docstring is also auto-generated from each action's `help_text` + `examples` + the `doc_sections` list above.

### Parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `action` | `str` | **Yes** | `""` | One of `run` \| `race` \| `pipeline`. Empty string returns `{"status": "error", "error": "action is required (pipeline \| race \| run)"}`. Unknown values return `"Unknown action 'X'. Use: pipeline \| race \| run"`. Case-insensitive (lowercased internally). |
| `tasks` | `list[dict]` | **Yes** | `[]` | Each dict: `{"name": "tool_name", "args": {...}, "feed"?: ...}`. Minimum 1 item. The `feed` key is only meaningful for `action="pipeline"` (see [Pipeline feed mechanism](#-pipeline-feed-mechanism)). **BREAKING v1.0**: renamed from `tools` (old name fails FastMCP schema validation). |
| `max_workers` | `int` | No | `4` | Thread pool size. Clamped to 1–8. Only meaningful for `run` and `race` (pipeline is sequential). |
| `allow_unsafe` | `bool` | No | `False` | If `True`, bypass `PARALLEL_SAFE` check. Only meaningful for `run` and `race` (pipeline is sequential — no concurrency hazard regardless). |
| `timeout` | `int` | No | `-1` | Per-call timeout override (seconds). `-1` → `cfg.worker_timeout` (default 60s); any `>=0` int → explicit override; other negatives also fall back to `cfg.worker_timeout` defensively. |
| `trace_id` | `str` | No | `""` | Trace identifier for logging and result correlation. Threaded into every per-task result and every error entry. |

## ⚡ Task Spec Format

```python
{"name": "web", "args": {"action": "search", "query": "..."}}
```

| Key | Type | Required | Description |
|-----|------|----------|-------------|
| `name` | `str` | **Yes** | Tool name. Must exist in `_TOOL_MAP` (17 tools). `"parallel"` is intentionally omitted (nested-parallel guard). |
| `args` | `dict` | No | Keyword arguments passed to the tool function. Default: `{}`. Must be a dict — non-dict values return `"Task spec at index N args must be a dict"`. |
| `feed` | `None` \| `str` \| `dict` | No | **Pipeline only.** Controls how the previous task's result flows into this task's args. See [Pipeline feed mechanism](#-pipeline-feed-mechanism). Ignored by `run` and `race`. |

## 🎬 Actions

### `action="run"` — Execute all in parallel, wait for all (barrier)

Submit all tasks to a `ThreadPoolExecutor`, wait for all to complete via `concurrent.futures.wait()`. Failures in individual tasks do NOT abort the run — they're returned in `errors`.

**Params used:** `tasks`, `max_workers` (clamped 1–8), `allow_unsafe`, `timeout`, `trace_id`.

**Returns:** `ok({"results": [...], "errors": [...], "completed": int, "failed": int, "duration_ms": int})`

**Per-task result entry:** `{"tool": str, "status": str, "result": Any, "trace_id"?: str}`
**Per-task error entry:** `{"tool": str, "error": str, "trace_id"?: str}`

This is the closest analogue to the pre-v1 `parallel()` behaviour.

**Examples:**
```python
# Parallel web searches
parallel(action="run", tasks=[
    {"name": "web", "args": {"action": "search", "query": "Python 3.12 features"}},
    {"name": "web", "args": {"action": "search", "query": "Rust async patterns"}},
])

# Parallel file reads with explicit per-call timeout
parallel(action="run", tasks=[
    {"name": "file", "args": {"action": "read", "path": "config.yaml"}},
    {"name": "file", "args": {"action": "read", "path": "README.md"}},
], timeout=30)

# Mixed safe tools with more workers
parallel(action="run", tasks=[
    {"name": "web", "args": {"action": "search", "query": "ChromaDB best practices"}},
    {"name": "python", "args": {"mode": "run", "code": "print(2 + 2)"}},
    {"name": "notify", "args": {"action": "send", "message": "Research started"}},
], max_workers=8)
```

### `action="race"` — First successful result wins

Submit all tasks in parallel; the first task to return a **non-error** result wins. Remaining tasks are cancelled if they haven't started yet (already-running tasks cannot be preempted — Python threads are not cancellable mid-execution).

**Params used:** `tasks`, `max_workers` (clamped 1–8), `allow_unsafe`, `timeout`, `trace_id`.

**Returns:** `ok({"winner": {...} | None, "cancelled": [str], "failed": [...], "duration_ms": int})`

The envelope status is `"success"` even when all tasks fail — the race itself completed; callers must check `winner is not None` to distinguish "race ran" from "race produced a winner".

**Use cases:**
- Send the same query to multiple search endpoints, use whichever returns first.
- Redundant fetches: hit a primary + fallback URL concurrently, take the first that succeeds.

**Examples:**
```python
# Race web vs tavily — first to return wins
parallel(action="race", tasks=[
    {"name": "web", "args": {"action": "search", "query": "Python async patterns"}},
    {"name": "tavily", "args": {"query": "Python async patterns"}},
], allow_unsafe=True)  # tavily is NOT in PARALLEL_SAFE

# Redundant fetch — primary + fallback URL
parallel(action="race", tasks=[
    {"name": "web", "args": {"action": "fetch", "url": "https://primary.example.com/data"}},
    {"name": "web", "args": {"action": "fetch", "url": "https://fallback.example.com/data"}},
])
```

---

### `action="pipeline"` — Sequential chain, results feed forward

Execute tasks **sequentially** (NOT parallel despite the tool name). Each task's result may be **fed** into the next task's args via the per-task `feed` key. Pipeline stops on first failure (no upstream result to feed downstream).

**Params used:** `tasks`, `timeout`, `trace_id`. (`max_workers` is NOT accepted — pipeline is sequential. `allow_unsafe` is accepted for API symmetry but ignored — pipeline has no concurrency hazard.)

**Returns:** `ok({"results": [...], "errors": [...], "completed": int, "failed": int, "duration_ms": int})` — same shape as `run` for uniformity.

**Pipeline does NOT enforce `PARALLEL_SAFE`** — tasks run sequentially so there is no concurrency hazard regardless of which tool is called.

#### Examples

```python
# Fetch web content, then run it through python — feed.text replaces args entirely
parallel(action="pipeline", tasks=[
    {"name": "web", "args": {"action": "fetch", "url": "https://example.com/api"}},
    {"name": "python", "args": {}, "feed": "result.text"},  # next call's args = result.text dict (must be dict)
])

# Read file, then consult on it — feed dict merges code into args
parallel(action="pipeline", tasks=[
    {"name": "file", "args": {"action": "read", "path": "bug_report.txt"}},
    {"name": "consult", "args": {"action": "review", "question": "What's the root cause?"}, "feed": {"context": "result.text"}},
])

# No feed — each step uses its own args (rare in pipelines but supported)
parallel(action="pipeline", tasks=[
    {"name": "git", "args": {"operation": "status"}},
    {"name": "notify", "args": {"action": "send", "message": "Git status checked"}},
])
```

## 🔗 Pipeline Feed Mechanism

The `feed` key on a pipeline task spec controls how the previous task's result flows into this task's args. Three modes:

| `feed` value | Behaviour | When to use |
|--------------|-----------|-------------|
| **`None`** (or omitted) | Use the task's own `args` as-is. No feeding. | When a step doesn't depend on the previous result (rare in pipelines but supported — e.g. a notify step that just signals "step N done"). |
| **`str`** (dot-path) | Resolve the dot-path against the previous result; the resolved value **REPLACES** the next call's args entirely. Only valid if the resolved value is a dict — otherwise the chain breaks with `"Pipeline feed 'X' did not resolve to a dict (got <type>)"`. | When the previous result IS the args dict for the next call (e.g. an LLM call that returns `{"action": "run", "code": "..."}` to be passed verbatim to `python`). |
| **`dict`** (arg-name → dot-path) | Keep the task's own `args` as the base, then **MERGE** each fed value in (overriding matching keys). Missing paths yield `None` (soft error — chain continues, downstream may tolerate None). | When you want to feed specific fields while keeping the rest of the args. Most common case. |

### Dot-path resolution (`_resolve_dot_path`)

The dot-path resolver supports:
- **Dict keys**: `"result.text"` against `{"result": {"text": "x"}}` → `"x"`
- **Object attrs**: same syntax against `obj.result.text` → `obj.result.text`
- **Multi-segment paths**: `"data.user.name"` traverses nested dicts/attrs
- **Missing segments** → `None` (defensive — caller decides whether None is fatal)
- **Empty path** (`""`) → returns the previous result unchanged

### Feed semantics — worked example

```python
# Previous result from a file read:
# {"status": "success", "result": {"text": "print('hello')", "lines": 1}, "trace_id": "abc"}

# Task spec 1 — str feed (REPLACE args):
{"name": "python", "args": {}, "feed": "result"}
# → next call runs with args = {"text": "print('hello')", "lines": 1}
#   (only valid because result is a dict; otherwise breaks the chain)

# Task spec 2 — dict feed (MERGE into args):
{"name": "python", "args": {"action": "run"}, "feed": {"code": "result.text"}}
# → next call runs with args = {"action": "run", "code": "print('hello')"}

# Task spec 3 — dict feed with missing path (soft error):
{"name": "python", "args": {"action": "run"}, "feed": {"code": "result.nonexistent"}}
# → next call runs with args = {"action": "run", "code": None}
#   (chain continues; python tool will reject `code=None` with its own validation error)
```

## 🔒 Security

### PARALLEL_SAFE Allowlist (10 tools)

```python
PARALLEL_SAFE = frozenset({
    "web",           # Network I/O — safe (each call has its own httpx client)
    "file",          # Read ops — safe (write ops have internal locks)
    "python",        # Sandboxed execution — safe (each call has isolated namespace)
    "python_exec",   # Alias for python — safe
    "notify",        # Desktop notification — safe (one-shot send)
    "github",        # API-only actions safe; push/pull excluded inside the github tool itself
    "consult",       # Cloud LLM call — stateless, safe to fan out (NEW v1.0)
    "vision",        # Cloud LLM call — stateless, safe to fan out (NEW v1.0)
    "report",        # Pure HTML/PDF generation — no shared mutable state (NEW v1.0)
    "agent",         # Specialist LLM dispatch — stateless per call (NEW v1.0)
})
```

**Excluded tools (and why):**

| Tool | Reason |
|------|--------|
| `git` | Write ops (`commit`, `push`) cause `index.lock` collisions |
| `memory` | ChromaDB concurrent writes risk `database is locked` |
| `cli` | Shell commands may conflict (e.g., two `mkdir` on same path) |
| `browser` | Playwright session — single browser context, concurrent actions corrupt state |
| `tavily` | Shared `AsyncTavilyClient` instance — concurrent calls race on internal state |
| `swarm` | Uses `ThreadPoolExecutor` internally — nesting risks thread-pool exhaustion |
| `workflow` | Long-running blocking calls — would hold pool workers for minutes |
| `parallel` | Nested parallel calls blocked by `_parallel_depth` guard (also omitted from `_TOOL_MAP` entirely) |

### `_TOOL_MAP` (17 tools, all lazy)

| Name | Maps To | In PARALLEL_SAFE? |
|------|---------|-------------------|
| `web` | `tools.web.web` | ✅ Yes |

| `file` | `tools.file.file` | ✅ Yes |

| `consult` | `tools.consult.consult` | ✅ Yes (NEW v1.0) |
| `vision` | `tools.vision.vision` | ✅ Yes (NEW v1.0) |

### Override (Use with Caution)

```python
parallel(action="run", tasks=[{"name": "tavily", "args": {"query": "..."}}], allow_unsafe=True)
```

This bypasses the `PARALLEL_SAFE` check. **Only use if you understand the risks** (e.g., all calls are read-only, or tools have internal locking). For `action="pipeline"`, `allow_unsafe` is accepted for API symmetry but ignored (sequential execution has no concurrency hazard).

## ⏱️ Timeout Enforcement

| `timeout` value | Effective timeout |
|-----------------|-------------------|
| `-1` (default) | `cfg.worker_timeout` (from `.env` `WORKER_TIMEOUT`, default 60s) |
| `0` | 0 seconds — `wait()` returns immediately; all futures marked as timed out (useful for "fire and forget" testing) |
| `>0` | Explicit per-call override (seconds) |
| `<-1` | Falls back to `cfg.worker_timeout` defensively (same as `-1`) |

**For `run`:** `concurrent.futures.wait(futures, timeout=effective_timeout)` enforces a true global deadline. Done futures are collected; not-done futures are marked `"Timed out after {effective_timeout} seconds"`.

**For `race`:** `concurrent.futures.as_completed(futures, timeout=effective_timeout)` — if no future completes within the timeout, `TimeoutError` fires and all unfinished futures are marked timed out.

**For `pipeline`:** the timeout applies per-step (each `fn(**args)` call). There is no global deadline across the whole chain — a 5-step pipeline with `timeout=30` can take up to 150 seconds.

**Why `wait()` instead of `as_completed()` for `run`?** The previous implementation used `as_completed()` + `future.result(timeout=30)`, which is broken: `as_completed()` blocks indefinitely waiting for a future to finish, so the per-future timeout never fires if the future hangs. `wait()` enforces a true global deadline. `race` still uses `as_completed()` legitimately — it needs to react to completion order.

**Note:** `ThreadPoolExecutor` cannot forcefully kill a hung thread. Timed-out threads remain orphaned until their internal timeout (e.g., `httpx` timeout, `subprocess` timeout) fires. This is acceptable for I/O-bound tools with their own timeouts.

## 🔄 Nested-Call Guard

The executor uses `threading.local()` to track recursion depth:

```python
_parallel_depth = threading.local()

def dispatch_run(...):
    if getattr(_parallel_depth, "value", 0) > 0:
        return fail("Nested parallel calls are not allowed", trace_id=trace_id)

    _parallel_depth.value = getattr(_parallel_depth, "value", 0) + 1
    try:
        # Execute calls
    finally:
        _parallel_depth.value -= 1
```

All three engines (`dispatch_run`, `dispatch_race`, `dispatch_pipeline`) increment the same thread-local. A `pipeline` stage that calls `parallel(action="run", ...)` is blocked the same way a `run` task that calls `parallel(action="race", ...)` is.

**Why?** If the LLM calls `parallel` inside `parallel`, it creates a thread-pool-of-thread-pools deadlock risk (outer waits for inner, inner waits for outer's pool). The guard prevents this with a clear error message.

## 🚨 Error Handling

| Error source | Where raised | Envelope `status` | Example |
|--------------|-------------|-------------------|---------|
| Empty `action` | Facade | `"error"` | `"action is required (pipeline \| race \| run)"` |
| Unknown `action` | Facade | `"error"` | `"Unknown action 'X'. Use: pipeline \| race \| run"` |
| `tasks` not a list | Action handler validator | `"error"` | `"tasks must be a list"` |
| Empty `tasks` | Action handler validator | `"error"` | `"No tasks provided"` |
| Task spec not a dict | Action handler validator | `"error"` | `"Task spec at index N must be a dict"` |
| Task spec missing `name` | Action handler validator | `"error"` | `"Task spec at index N missing 'name'"` |
| Task `args` not a dict | Action handler validator | `"error"` | `"Task spec at index N args must be a dict"` |
| Tool not in `_TOOL_MAP` | Action handler validator (`_get_tool_fn` returns `None`) | `"error"` | `"Tool 'X' not found"` |
| Tool not in `PARALLEL_SAFE` (run/race only) | Action handler | `"error"` | `"Tool 'X' is not parallel-safe. Set allow_unsafe=True to override."` |
| Pipeline `feed` invalid type | `pipeline.py` validator | `"error"` | `"Task spec at index N feed must be str \| dict \| None (got <type>)"` |
| Pipeline `feed` str didn't resolve to dict | `dispatch_pipeline` | `"success"` (with `errors`) | `"Pipeline feed 'X' did not resolve to a dict (got <type>)"` — chain breaks, downstream steps not executed |
| Pipeline `feed` dict dot-path invalid type | `dispatch_pipeline` | `"success"` (with `errors`) | `"Pipeline feed for arg 'X' must be a dot-path string"` — chain breaks |
| Per-task exception (run/race) | `dispatch_run` / `dispatch_race` | `"success"` (with `errors`) | `"<ExceptionType>: <message>"` — recorded in `errors`, other tasks continue |
| Per-task exception (pipeline) | `dispatch_pipeline` | `"success"` (with `errors`) | `"<ExceptionType>: <message>"` — chain breaks (no upstream result to feed downstream) |
| Nested parallel call | Executor (`_parallel_depth` guard) | `"error"` | `"Nested parallel calls are not allowed"` |
| Per-task timeout (run/race) | `dispatch_run` / `dispatch_race` | `"success"` (with `errors`) | `"Timed out after {effective_timeout} seconds"` |
| Handler raises unexpected exception | Facade try/except | `"error"` | `"Parallel action failed: <exception message>"` |
| Handler returns non-dict | Facade | `"error"` | `"Handler returned <type>, expected dict."` |

### Per-action error semantics summary

| Action | On per-task failure | On timeout | On `PARALLEL_SAFE` violation |
|--------|---------------------|-----------|------------------------------|
| `run` | Continue — error in `errors[]`, other tasks finish | Other tasks continue; timed-out tasks in `errors[]` | Whole call returns `fail()` before any task runs |
| `race` | Continue — failed task in `failed[]`, race continues until first success or all fail | All unfinished futures marked timed out in `failed[]`; `winner` may be `None` | Whole call returns `fail()` before any task runs |
| `pipeline` | **Abort chain** — error in `errors[]`, downstream steps not executed | Per-step (each `fn(**args)` call); no global deadline across chain | N/A — pipeline does not enforce `PARALLEL_SAFE` |
